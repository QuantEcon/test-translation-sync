# test-translation-sync

Test harness for [QuantEcon/action-translation](https://github.com/QuantEcon/action-translation). This repo is the **English source**; its translated targets are [test-translation-sync.zh-cn](https://github.com/QuantEcon/test-translation-sync.zh-cn) and [test-translation-sync.fa](https://github.com/QuantEcon/test-translation-sync.fa).

Nothing here is published. The lectures are fixtures.

## Why it exists

To exercise an engine release through **the same wiring production runs** before it reaches a live edition.

Every workflow in the three repos uses `uses: QuantEcon/action-translation@v0` — the published floating tag, exactly as the QuantEcon lecture editions do. That is deliberate, and it is the whole point: the harness catches *config* regressions as well as code regressions. A workflow that checks the action out and builds it (`uses: ./action`) proves the code compiles and tells you nothing about whether `@v0` resolves to the release you think the estate is running — which is the change that most often goes wrong. See [action-translation#109](https://github.com/QuantEcon/action-translation/issues/109).

To test **unreleased** code, drive the CLI locally against these repos with a built bundle. Do not pin a harness workflow to a branch: `uses:` accepts no expression, so a workflow cannot switch versions dynamically, and pinning defeats the property above.

## Running a test

The 26 open PRs on this repo are a standing fixture suite — one per diff shape the engine has to survive:

| Range | Covers |
|---|---|
| 01–08 | intro/title/content edits, section reorder, add, delete, subsection change, multi-element |
| 09–16 | real-world lecture update, sub-subsection add/change/delete, code cells, display math, pure reorder |
| 17–20 | TOC changes — new document, deleted document, multi-file, rename |
| 21–26 | frontmatter only, deep nesting, special characters, empty sections, pre-title content, heading case |

**Add the `test-translation` label to a fixture PR.** The sync workflow runs in test mode (using the PR head rather than a merge commit), translates into both targets, and opens a translation PR on each — which then draws a real review from `@v0`. Nothing needs to be merged, and `main` is not touched.

Merging a PR here instead runs the same sync in production mode. Commenting `\translate-resync` on a merged PR re-triggers it.

The `test/*` branches are the test program itself. **Do not delete them.**

## Resetting

A test run leaves residue — merged fixture PRs, merged sync PRs on the targets, drifted content. The baseline for all three repos is the `harness-baseline` tag, and the reset commands live in the program's command bench, [QuantEcon/project-translation](https://github.com/QuantEcon/project-translation):

    make harness                      # where each baseline sits, and whether content drifted
    make harness-reset ARGS=--dry-run # what a reset would restore
    make harness-reset                # restore content + .translate/ to baseline
    make harness-reset ARGS=--all     # also close open translation PRs, delete sync branches

Reset restores **content and `.translate/` metadata only — never `.github/`**. Workflows must stay current, not baseline; rolling them back to whatever they were when the tag was cut would silently undo the "mirror production" property this repo exists for.

Move the baseline (`make harness ARGS=--cut`) only when the baseline *content* is meant to change — a new fixture lecture, a corrected translation, re-bootstrapped state. Workflow changes never require it.

## Layout

    lecture.md           fixture lecture — nested sections, code cells, display math
    lecture-minimal.md   fixture lecture — small, for fast structural tests
    _toc.yml             table of contents

Targets additionally carry `.translate/` (config and per-file state), generated by the engine with `translate status --write-state` — never hand-written. The key names are validated strictly and a mismatch is ignored *silently*, so a hand-edited state file reads as no state at all.

Program-level notes: `infrastructure/harness.md` in project-translation.
